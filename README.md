# Facebook Group Scraper

Node.js and Playwright scraper for Facebook group content, built to run from the same codebase locally, in Docker, or inside Apify.

The project is optimized around a network-first extraction path, persistent run state, and structured outputs that are usable both for scraping workflows and downstream analysis. It supports:

- authenticated cookie-backed sessions for content already visible in the source browser session
- public-group scraping without cookies
- proxy and proxy-pool execution
- resume/checkpoint behavior
- group-specific output folders
- Gemini-powered BI-style XLSX generation

## Capability Matrix

| Mode | Supported Surface | Notes |
| --- | --- | --- |
| Direct mode, no cookies | Public groups only | Intended for public-group scraping without authenticated browser state |
| Cookie-backed authenticated session | Content visible to that browser session | Uses `cookies.json` or storage-state cookies; useful when the session can already access additional group content in a normal browser |
| Proxy mode | Public or authenticated session paths | Supports single proxies, proxy pools, retries, quarantine, and Apify proxy integration |
| Analyzer mode | `output/<groupId>/output.json` | Produces a BI-oriented `output.xlsx` in the same group folder |

## Architecture

The runtime is centered around one scraping loop with explicit extraction priorities:

1. **Browser and session bootstrap**
   - launches Playwright
   - optionally loads cookies from `cookies.json`
   - optionally restores per-proxy session state
   - warms up Facebook before group navigation

2. **Structured extraction first**
   - inspects embedded document JSON
   - captures and parses relevant Facebook network payloads
   - normalizes posts from structured sources before considering DOM fallback

3. **DOM fallback only when needed**
   - advances the feed conservatively
   - extracts visible post cards only after structured extraction stops producing useful new posts
   - recovers automatically if a late-stage feed action navigates into a post permalink

4. **Persistence and recovery**
   - deduplicates posts continuously
   - writes outputs per group under `output/<groupId>/`
   - saves checkpoints, stats, logs, payload samples, and session-state artifacts

5. **Optional analysis layer**
   - reads the group’s `output.json`
   - asks Gemini to infer a group-specific schema and English-normalized values
   - writes a BI-oriented workbook back into the same folder

## Output Contract

Each group is written into its own folder:

```text
output/<groupId>/
  output.json
  posts.json
  posts.jsonl
  checkpoint.json
  stats.json
  output.xlsx
  logs/
    run.log
  debug/
  session-state/
```

Key files:

- `output.json`: lightweight public export used by the analyzer and downstream pipelines
- `posts.json`: full normalized archive of accepted posts
- `posts.jsonl`: line-delimited version of the same accepted post archive
- `checkpoint.json`: persisted resume state
- `stats.json`: run-level metrics and extraction quality counters
- `output.xlsx`: BI-oriented workbook generated by the analyzer
- `logs/run.log`: structured runtime log
- `debug/`: payload samples, failure metadata, and proxy diagnostics
- `session-state/`: persisted anonymous/authenticated browser state when enabled

## Authentication Model

This project does **not** implement login automation, CAPTCHA solving, or account creation flows.

Instead, it supports two operating modes:

- **Without cookies**: public-group scraping only
- **With cookies**: cookie-backed authenticated browser sessions for content already visible to that session in a normal browser

That means the scraper can use a `cookies.json` export from an existing browser session, but it does not try to sign in for you. The conservative assumption is:

- if the session can already see the group content in a browser, the scraper can use that same session state
- if no authenticated cookies are provided, the supported mode is public groups only

## Quick Start

Install dependencies:

```bash
npm install
npx playwright install chromium
```

Create your env file:

```bash
cp .env.example .env
```

Minimal `.env`:

```env
GROUP_URL=https://www.facebook.com/groups/123456789012345/
MAX_POSTS=100
RUNTIME_MINUTES=
COOKIES_FILE=
HEADLESS=true

PROXY_SERVER=
PROXY_USERNAME=
PROXY_PASSWORD=
PROXY_POOL_DIR=
PROXY_POOL_PROTOCOL=

GEMINI_API_KEY=
```

Run with `.env` defaults:

```bash
npm start
```

If the group URL is `https://www.facebook.com/groups/123456789012345/`, outputs are written to:

```text
output/123456789012345/
```

## Common Workflows

Scrape with current `.env` defaults:

```bash
node src/index.js --max-posts 100
```

Scrape one-off public content directly:

```bash
node src/index.js --url "https://www.facebook.com/groups/..." --no-proxy --max-posts 100
```

Run the API server:

```bash
npm run start:api
```

Request a scrape and save the returned `output.json`:

```bash
curl -X POST http://127.0.0.1:3000/scrape \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $API_AUTH_KEY" \
  -d '{"groupUrl":"https://www.facebook.com/groups/.../","maxPosts":25}' \
  -o output.json
```

Scrape using an authenticated cookie export:

```bash
node src/index.js --cookies-file ./cookies.json --max-posts 100
```

Verify the configured proxy path:

```bash
node src/index.js --test-proxy
```

Run scheduled scraping and analyze afterward:

```bash
node src/index.js \
  --schedule-total-minutes 120 \
  --schedule-interval-minutes 15 \
  --schedule-run-analyzer=true
```

Generate the workbook for an existing group folder:

```bash
npm run analyze:xlsx
```

Or target a specific group explicitly:

```bash
node src/analyze/index.js --group-url "https://www.facebook.com/groups/123456789012345/"
```

## Cookies

Cookie loading is supported through:

- `COOKIES_FILE=./cookies.json`
- `--cookies-file ./cookies.json`

Accepted formats:

- a plain JSON cookie array
- a Playwright-style storage-state object with top-level `cookies`

When cookies are present, the runtime keeps the browser session sticky and disables the anonymous recycle strategy that is used in no-cookie mode.

## API

The API is intentionally small and synchronous:

- `GET /health`: readiness and busy state
- `POST /scrape`: accepts `groupUrl` and `maxPosts`, runs a scrape, and returns the generated `output.json`

Required API env vars:

- `API_AUTH_KEY`
- `API_ALLOWED_ORIGIN`

Optional API env vars:

- `API_COOKIES_FILE`

Example request body:

```json
{
  "groupUrl": "https://www.facebook.com/groups/123456789012345/",
  "maxPosts": 25
}
```

Behavior notes:

- the API forces direct mode with `--no-proxy`
- the API forces `HEADLESS=true`
- the API defaults to no cookies, but can use a server-side cookies file when `API_COOKIES_FILE` is set
- `POST /scrape` requires `X-API-Key: <secret>`
- `GET /health` remains public
- browser requests are allowed only from the exact `API_ALLOWED_ORIGIN`
- only one scrape request runs at a time; concurrent requests receive HTTP `409`

Example local request:

```bash
curl -X POST http://127.0.0.1:3000/scrape \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $API_AUTH_KEY" \
  -d '{"groupUrl":"https://www.facebook.com/groups/123456789012345/","maxPosts":25}' \
  -o output.json
```

Browser apps must send:

- `Origin: https://your-frontend.example.com`
- `X-API-Key: <secret>`

Important security note:

- if you embed `X-API-Key` in a public browser app, users can extract and reuse it
- for untrusted public users, put your own backend in front of this API instead of exposing the key to the browser

If you need authenticated group access on the hosted API:

- place `cookies.json` in the project root before `docker compose build`
- set `API_COOKIES_FILE=cookies.json` in `.env`
- rebuild the image after changing the cookie file

## Gemini Analyzer and BI Workbook

The analyzer reads the group’s `output.json`, infers a group-aware schema, translates and normalizes extracted values into English, and writes a BI-friendly workbook into the same group folder.

Current workbook structure:

- **main analysis sheet**: fixed analytics columns plus group-specific dynamic columns
- **`group_info`**: workbook metadata, group type, summary, and row count
- **`data_dictionary`**: column inventory with key, label, type, and source

Fixed workbook columns include:

- post URL
- created timestamp
- calendar week
- weekday
- group type
- profile name
- source language
- original post text
- English translation
- English analysis summary
- analysis confidence
- reactions, comments, and shares

This makes the workbook more suitable for filtering, pivoting, BI ingestion, and dashboard work than a free-form AI summary sheet.

## Public Hosting

Recommended production shape on Debian:

- Docker Compose runs the `api` container on `127.0.0.1:3000`
- host-level Nginx terminates TLS on your API subdomain
- only ports `80` and `443` are publicly exposed
- port `3000` stays private to localhost

1. Set the API env vars in `.env`:

```env
API_HOST=0.0.0.0
API_PORT=3000
API_AUTH_KEY=replace-with-a-long-random-secret
API_ALLOWED_ORIGIN=https://app.example.com
API_COOKIES_FILE=cookies.json
```

2. Start the API container:

```bash
DOCKER_UID=$(id -u) DOCKER_GID=$(id -g) docker compose up -d --build api
curl http://127.0.0.1:3000/health
```

3. Install Nginx and Certbot on Debian:

```bash
sudo apt update
sudo apt install -y nginx certbot python3-certbot-nginx
```

4. Install the checked-in Nginx templates:

```bash
sudo cp deploy/nginx/scrape-api-limit.conf /etc/nginx/conf.d/scrape-api-limit.conf
sudo cp deploy/nginx/facebook-group-scraper-api.conf /etc/nginx/sites-available/facebook-group-scraper-api.conf
```

5. Edit the server name in the site file to match your subdomain, then enable it:

```bash
sudo ln -sf /etc/nginx/sites-available/facebook-group-scraper-api.conf /etc/nginx/sites-enabled/facebook-group-scraper-api.conf
sudo nginx -t
sudo systemctl reload nginx
```

6. Issue the TLS certificate after DNS points to the server:

```bash
sudo certbot --nginx -d api.example.com
sudo nginx -t
sudo systemctl reload nginx
```

7. Keep the firewall limited to `80/443`:

```bash
sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
```

Smoke tests:

```bash
curl https://api.example.com/health
curl -X POST https://api.example.com/scrape \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $API_AUTH_KEY" \
  -d '{"groupUrl":"https://www.facebook.com/groups/.../","maxPosts":25}' \
  -o output.json
```

Deployment requirements:

- DNS for the subdomain must point to the server before issuing TLS
- keep `API_ALLOWED_ORIGIN` set to the exact frontend browser origin, not the API subdomain unless they are the same
- do not open port `3000` publicly; the Compose file binds it to `127.0.0.1` only

## Apify

The repository includes an Apify Actor wrapper and Actor definition files:

- `.actor/actor.json`
- `.actor/input_schema.json`

## Project Layout

- `src/config/`: env loading, CLI parsing, validation
- `src/browser/`: Playwright session boot, cookies, proxies, session state
- `src/extract/`: network/document parsing and DOM fallback
- `src/core/`: crawl loop, normalization, deduplication, checkpoints, recovery
- `src/output/`: logs and persisted artifacts
- `src/analyze/`: Gemini analysis and XLSX generation
- `src/apify/`: Actor wrapper and Apify-specific behavior
- `test/`: regression coverage across parser, config, analyzer, proxy, and support logic

## Known Fragility Points

- Facebook changes markup and internal payloads regularly.
- Logged-out visibility varies across groups, sessions, IPs, and proxies.
- Proxy quality affects not just connectivity but feed depth and stability.
- DOM extraction is intentionally a fallback path, not the primary source of truth.
- Cookie-backed runs depend on the quality and continued validity of the source browser session.
